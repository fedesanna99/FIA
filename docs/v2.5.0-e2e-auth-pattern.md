# Auth pattern · v2.5.0-paolo-ready-smoke-e2e-live T1

**Data**: 2026-05-26
**Scope**: T1 brief — capire come l'app autentica l'utente, prima di scrivere il login helper in Playwright.

## Pattern

**Tipo**: form **email + password** classico. NO OAuth, NO magic link. STOP rule #1 non attivata.

## Componenti chiave

| File | Ruolo |
|------|-------|
| `frontend/src/main.tsx:22-25` | mount `<AuthGate><App/></AuthGate>` — l'intera App vive dentro l'AuthGate |
| `frontend/src/components/auth/AuthGate.tsx` | gate obbligatorio: se `user === null` mostra `<AuthScreen />`, altrimenti renderizza i children (l'App) |
| `frontend/src/components/auth/AuthScreen.tsx` | form login+register con tab switcher |
| `frontend/src/api/auth.ts` | `login()` + `register()` POST → backend `/api/auth/login` o `/api/auth/register` |
| `frontend/src/store/authStore.ts` (Zustand persist) | persisted state `{token, user}` in localStorage key `auth-store` |

## URL della pagina login

**NESSUN URL dedicato**. L'AuthScreen è un componente full-screen renderizzato sopra l'App quando `user === null`. L'URL della pagina resta `https://fea-pro.fly.dev/` (path `/`).

Implicazione test: NON aspettare un `waitForURL` cambio path post-login. Aspettare invece il **smontaggio dell'AuthScreen** (test-id `auth-submit` sparisce) o il **mount del Dashboard** (test-id `dashboard-root` appare).

## Selettori UI

| Elemento | Selettore robusto (test-id) | Selettore fallback (role) |
|----------|------------------------------|----------------------------|
| Input email | `[data-testid="auth-email"]` | `input[type="email"]` |
| Input password | `[data-testid="auth-password"]` | `input[type="password"]` |
| Submit button | `[data-testid="auth-submit"]` | `button[type="submit"]:has-text("Accedi")` |
| Tab login (in caso non sia attivo di default) | `[data-testid="auth-tab-login"]` | `button:has-text("Accedi")` (tab) |
| Errore login | `[data-testid="auth-error"]` (visibile solo se errore) | `[role="alert"]:has-text("...")` |

Default mode: **login** (il componente apre `mode === "login"` di default).

## Comportamento post-login

1. Form submit → `handleSubmit` (AuthScreen.tsx) chiama `apiLogin(email, password)`
2. Success → `authStore.setSession({token, user})` → `localStorage.setItem("auth-store", ...)`
3. AuthGate detecta `user !== null` → smonta AuthScreen, monta App
4. App montato → default route `/` → Dashboard.tsx renderizzato
5. Dashboard ha `data-testid="dashboard-root"` (anchor per attendere completamento login)

Failure → AuthScreen mostra `<div data-testid="auth-error">` con messaggio italiano da `translateApiError`.

## Anchor post-login per i test

| Anchor | Test-id | Note |
|--------|---------|------|
| Dashboard mounted | `[data-testid="dashboard-root"]` | preferito |
| QuickAct "Da template" | `[data-testid="dashboard-action-template"]` | per step 1 successivo |
| Topbar visibile | `[data-testid="topbar-save-chip"]` | fallback se Dashboard non si è caricato |

## Verdetto

Pattern email+password standard. Test-id `auth-*` permettono selettori robusti senza fragility da CSS classes o copy ITA/EN.

**STOP rule #1 NON attivata** (no OAuth-only).

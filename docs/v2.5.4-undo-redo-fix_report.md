## Comportamento BEFORE/AFTER

### Scenario 1 · Primo load (utente apre un modello dal dashboard)

- **BEFORE**: `setModel(m)` con `current === null` → clear + push(m). Past = 1.
- **AFTER**: branch `!isSameModel` → idem (clear + push). Past = 1. ✓ invariato

### Scenario 2 · Switch ad altro modello

- **BEFORE**: `setModel(m2)` con `current = m1` → clear + push(m2). Past = 1.
- **AFTER**: `current.id !== m2.id` → branch `!isSameModel` → idem. Past = 1. ✓ invariato

### Scenario 3 · addNode via NodeDialog (FIX vero)

- **BEFORE**:
  1. `addNode(n)` → `applyModelAndPush` → `history.push(m+n)`. Past = `[baseline, m+n]`. canUndo = true.
  2. `invalidateQueries(["model", id])` → refetch → `useModel.ts:24` → `setModel(refetched)`.
  3. `setModel` esegue `history.clear()` + `history.push(refetched)`. Past = `[refetched]`. **canUndo = false**.
  4. Ctrl+Z no-op. Nodo NON viene rimosso. **Bug osservato.**

- **AFTER**:
  1. `addNode(n)` → push. Past = `[baseline, m+n]`.
  2. Refetch → `setModel(refetched)`.
  3. `isSameModel` = true (`current.id === refetched.id`).
  4. `modelSnapshotsEqual(last, refetched)` = true (refetched coincide con `m+n` appena pushato). **No push, no clear**. Past resta `[baseline, m+n]`.
  5. canUndo = true. Ctrl+Z funziona, nodo rimosso. ✓ **FIX**

### Scenario 4 · Refetch divergente (backend ha normalizzato qualcosa)

- **BEFORE**: clear + push → past = 1. Si perde tutta la history precedente.
- **AFTER**: equals false → push aggiuntivo, history estesa di 1. ✓ **migliore** (history preservata anche in casi di divergenza server-side)

### Scenario 5 · Logout / reset (setModel(null))

- **BEFORE**: `set({model: null, lastSavedAt: null, ...})` + clear (implicito perché `if (m) history.push(m)` skippa con m null). Past = 0.
- **AFTER**: early-return esplicito con `m === null` → clear + niente push. Past = 0. ✓ invariato, ma più esplicito e leggibile.

### Scenario 6 · MeshWizardDialog (bonus, oggetto del brief secondario)

- **BEFORE**: re-mesh genera un nuovo modello con stesso `id` ma struttura nodi/elementi completamente diversa → `setModel(remeshed)` → clear + push. Perdi tutta la history pre-mesh.
- **AFTER**: `isSameModel` = true, ma `modelSnapshotsEqual` = false (lengths divergono) → push aggiuntivo. **Posso fare undo del re-mesh** (era impossibile prima del fix).

### Scenario 7 · EditModelDialog (rename modello)

- **BEFORE**: rename → backend POST/PATCH → `setModel(renamed)` → clear + push. Perdi tutta la history per cambiare il nome.
- **AFTER**: `isSameModel` = true; `modelSnapshotsEqual` confronta nodes/elements/loads/constraints **escludendo `name`** → equals = true → no push, no clear. La history pre-rename è preservata. L'utente può modificare-rename-undo senza perdere lo storico delle entità. Il rename in sé non è undoable (acceptable: rename è metadata, gli ingegneri si aspettano undo su entità geometriche).

## Backward compatibility

- Comportamento invariato per: primo load, switch modello, logout/reset.
- Comportamento **migliorato** per: addNode/Element/Load/Constraint via dialog (history preservata, undo funziona), re-mesh (undoable), import dello stesso modello (history preservata).

## Costo prestazionale

`modelSnapshotsEqual` esegue:
- 4 length checks (O(1))
- 4 `JSON.stringify` + string compare sui campi `nodes`/`elements`/`loads`/`constraints` (O(n) in dim modello, solo quando le lengths coincidono).

Per modelli reali Paolo-target (~100-500 entità), `JSON.stringify` di 4 array è dell'ordine di pochi ms — invisibile rispetto alla latency rete del refetch (~50-200ms).

Per modelli grandi (>10k entità), `JSON.stringify` può salire a 50-100ms. Acceptable per ora: il refetch stesso costerebbe altrettanto. Se diventa un problema in futuro, sostituire con un hash veloce (es. xxhash sui field id-ordinati).

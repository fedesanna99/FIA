### Sintesi operativa

- **Sintomo**: i 3 percorsi UI per lanciare l'export di un report (button "Inspect" nella ReportExportDialog, voce palette "wizard", voce palette "quick") **non producono nessun effetto observable**: nessun download, nessuna chiamata di rete, nessun toast d'errore, nessun cambiamento DOM. L'evento click non viene tradotto in una request al backend.
- **Formato broken**: entrambi PDF e XLSX, a livello UI. **Il backend funziona correttamente**: gli endpoint del router producono file validi (PDF 5160 byte 2-page leggibile, XLSX 8816 byte valid Excel). Si confermano OK anche DXF (18947B) e IFC4 (4096B).
- **Ambiente**: confermato sulla versione live https://fea-pro.fly.dev/ (v2.3.2). Non verificato esplicitamente in dev locale dall'audit, ma il codice frontend impattato è lo stesso bundle.
- **Repro steps documentati** (audit live):
  1. Aprire workspace con un modello caricato (es. uno dei demo `ex_simple_beam_2d`)
  2. Eseguire un'analisi statica (per avere risultati esportabili)
  3. Tentare tutti e 3 i path UI:
     - Click button "Inspect" nel pannello/dialog di export
     - Aprire command palette (Ctrl+K) → voce "wizard" export
     - Aprire command palette → voce "quick" export
  4. Osservare: **nessuna delle 3 strade scarica un file o produce un errore visibile**.
- **Stack trace / error**: non riportato. L'audit è black-box, ha solo registrato l'assenza di effetti observable (no DOM mutation, no overlay, no network request, no console error). Il termine usato è "event dispatcher rotto frontend": probabilmente i click handler dei 3 entry point UI non sono cablati alla funzione che esegue il fetch verso l'endpoint backend di export (oppure la chiamata avviene ma viene swallow-ata silenziosamente).
- **Diagnosi tecnica preliminare** (da incrociare con il futuro brief diag PDF): da indagare i file `frontend/src/components/dialogs/ReportExportDialog.tsx`, `frontend/src/utils/export.ts`, `frontend/src/utils/exportXlsx.ts`, `frontend/src/components/panels/ExportServerPanel.tsx`, e il binding sui voce-comandi della command palette. Il backend (`backend/core/io/pdf_report.py`, `backend/core/io/excel_export.py`, `backend/api/routes/io.py`) è confermato funzionante e non è stato modificato dal commit iniziale `e25ca88` (l'ultimo tocco su ReportExportDialog è stato cosmetico, commit `498e8bd` PR17 T9 Precision finalize).

> ⚠️ **Discrepanza fra le due fonti**: il report.md riga 80 marca Undo come ❌ (testato live con `addNode` reale → nodo non rimosso), mentre L2_functional_flows_report.md riga 51 marca un test "Ctrl+Z dopo modifica → state revertito" come ✅ senza console error. La spiegazione plausibile è che il test L2 Playwright preme Ctrl+Z senza prima fare una mutation che passi davvero dal flow modelStore (es. l'invoke avviene in stato vuoto o su un attributo che non popola la history): non c'è nulla da revertire e Ctrl+Z "non rompe nulla", il che il test interpreta come PASS. Il test live (audit report) invece esegue una vera `addNode` via dialog e osserva che il nodo **resta** dopo Ctrl+Z. **Conclusione**: bug confermato come descritto dal report; L2 produce un falso positivo perché non valida la mutation effettiva.

### Sintesi operativa

- **Sintomo**: dopo aver aggiunto un nodo (`addNode` via dialog UI), né la shortcut **Ctrl+Z** né il **button "Annulla"** in topbar producono l'undo. Il nodo aggiunto **rimane nel modello**. Stessa cosa per Redo (test N2 ❌ "dipende N1").
- **Flow scatenante**: aggiunta di un nodo via dialog UI (NodeDialog). Per estensione, plausibilmente ogni mutation che non passa per `modelStore.replaceModel`/`useModelHistory.getState().push(...)`. Lato store, il wiring è effettivamente presente in `frontend/src/store/modelStore.ts` (replaceModel + loadModel pushano in history, undo()/redo() leggono da history), ma il dialog NodeDialog probabilmente esegue la mutation tramite un'API diversa che bypassa il push.
- **Da quando broken**: la feature è stata introdotta in **v2.3.0** (commit `285a0a7` 2026-05-23 "feat(v2.3.0): Multi-model compare polish + Undo/Redo store-level"). L'audit live (2026-05-23/24) ha trovato il bug **subito dopo l'introduzione** della feature. Quindi **non è una regression**: la feature è **broken sin dalla nascita** sul lato UI (i test unitari su `modelStore` e `historyStore` passano, ma il wiring fra dialog → modelStore non è completo). Conferma collaterale da `BACKLOG.md` riga 518: "`useModelHistory` → history push wiring nei restanti store" è ancora tech-debt aperto v2.4.x.
- **Severity**: P0 (da `consolidated_bug_list.md` riga 127, classificato BLOCKER ship-stop nel report.md).
- **Test/repro documentati**: 
  - Manuale live (report.md riga 80-81): apri workspace → addNode via dialog → Ctrl+Z o click "Annulla" → osserva che il nodo è ancora presente.
  - Test unitari `frontend/src/store/historyStore.test.ts` e `frontend/src/store/modelStore.test.ts` (entrambi nei 584/584 PASS): **passano** perché esercitano direttamente `useModelStore.getState().replaceModel(...)` e `undo()`, scenari che funzionano. Il bug vive nello spazio fra dialog e store.

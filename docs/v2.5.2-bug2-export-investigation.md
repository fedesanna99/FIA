## 3. Path A В· Button "Genera report PDF" in `ResultsOverviewCard`

**Dispatch**: `frontend/src/components/shell/ResultsOverviewCard.tsx:141-149`

**Listener**: `frontend/src/App.tsx:289-291`


`reportExportOpen` ГЁ state locale di App (riga 170). Quando true, monta `<ReportExportDialog open={reportExportOpen} ... />` (riga 694) tramite lazy import (riga 79-81).

**Dialog handler**: `frontend/src/components/dialogs/ReportExportDialog.tsx:84-103`

```tsx
function handleDownload() {
  if (!model) { toast("warning", "Apri un modello..."); return; }
  setDownloading(true);
  try {
    generateReport({
      model,
      staticResults: staticRes ?? null,
      modalResults: modalRes ?? null,
    });
    notify("success", "Report PDF generato", `${pageCount} sezioni В· ${model.name}`);
    setDownloading(false);
    onClose();
  } catch (e) {
    setDownloading(false);
    toast("error", `Errore generazione PDF: ${(e as Error).message}`);
  }
}
```

**Cosa fa oggi**:
- Click вҶ’ dispatch event (sincrono, immediato)
- Listener in App.tsx вҶ’ setReportExportOpen(true) вҶ’ dialog modale si apre con checklist 5 sezioni
- User selects sections, clicca "Scarica PDF" вҶ’ `generateReport()` produce file e `doc.save()` triggera download nel browser
- toast success "Report PDF generato"

**Causa probabile dell'audit "morto"**: 
- (i) Forse l'event listener era stato rimosso/non registrato in v2.3.2 e ripristinato in seguito (verificare git blame su App.tsx:291 vs data audit).
- (ii) Forse il dialog si apriva ma il button "Scarica PDF" era disabilitato perchГ© `!staticResults` (riga 175 `disabled={!model || !staticRes || downloading}`) e l'utente percepiva il dialog stesso come "morto".
- (iii) Possibile race condition se l'event veniva dispatcato prima che l'effect di App.tsx registrasse il listener (improbabile, dato che addEventListener ГЁ in useEffect mount).

**Status stimato attuale**: probabilmente **funzionante** (event вҶ’ dialog вҶ’ button вҶ’ generateReport tutto in place). Da verificare con repro live nel brief v2.5.3.

## 7. Caller funzionanti вҖ” pattern di riferimento

`ExportView` (rail destro Tools вҶ’ Export) ГЁ il caller **chiaramente funzionante** dei dispatcher client-side. Pattern:

1. Hook button click вҶ’ handler async dedicato (`doPdf`, `doExcel`, ecc.)
2. `setBusy(...)` per disabilitare button durante operazione
3. `try { await ... } catch { toast error } finally { setBusy(null) }`
4. **Toast esplicito di success al termine** (`"Report PDF generato."`, `"XLSX scaricato (N sheet: ...)"`)

`Toolbar.tsx:131` e `AvatarMenu.tsx:88` chiamano `generateReport` similmente con feedback.

Ricetta che gli altri path possono adottare: dopo `generateReport()` mettere sempre `notify("success", ...)` o `toast("success", ...)`. Questo ГЁ il **fix minimo** che probabilmente porta a "morto" вҶ’ "funzionante percepito" senza toccare il dispatcher.

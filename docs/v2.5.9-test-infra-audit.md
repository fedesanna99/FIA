## L2 falso positivo (target T3)

**Location**: `frontend/e2e/quality-checkpoint/L2_functional_flows/flow_compare_undo.spec.ts:16-42`

**Test attuale**:
```ts
test("happy: Ctrl+Z dopo modifica → state revertito", async ({ page }) => {
  await authedGoto(page, "/?model=ex_simple_beam_2d");
  // ...
  await page.keyboard.press("Control+Z");
  await page.waitForTimeout(500);
  // Verifica: nessun errore console + HUD invariato
  expect.soft(consoleErrs.length, ...).toBe(0);
  // BUG-9 (audit interno): Undo/Redo v2.3.0 broken in produzione
  // Documentiamo che il test sopravvive ma è da rivedere quando si fixa
});
```

**Confessione del falso positivo nei commenti**: riga 40-41 documenta esplicitamente che il test "sopravvive ma è da rivedere quando si fixa". **Sopravvive perché esercita Ctrl+Z su modello appena caricato senza modifiche → no-op atteso, no console errors → asserisce solo "no errori" non "state revertito"**. Il titolo del test ("dopo modifica → state revertito") mente sul contenuto.

**Fix v2.5.4** (sprint passato): bug #9 chiuso. `modelStore.setModel` ora preserva history su refetch stesso model.id. Quindi il test L2 può essere riscritto per esercitare davvero il flow:
1. carica modello
2. aggiungi nodo via dialog (passa per useMutation + invalidateQueries + refetch)
3. attende network idle (refetch completato)
4. Ctrl+Z
5. assert: count nodi tornato al valore pre-add

**Decisione T3**: il test `flow_compare_undo.spec.ts:16-42` va aggiornato. Il vero test reale che esercita il flow deve essere o (a) un test riscritto nello stesso spec, o (b) marcato `.fixme()` con pointer a un nuovo test in altro spec. Decisione: **(a) riscrittura nello stesso spec** mantenendo il titolo (`"happy: Ctrl+Z dopo modifica → state revertito"`) ma con asserzioni che davvero verificano il revert.

**Caveat backend**: il test richiede backend FastAPI running per fare la mutation. In CI questo è OK (CI può accendere backend). In esecuzione locale richiede setup. Documento la dipendenza.

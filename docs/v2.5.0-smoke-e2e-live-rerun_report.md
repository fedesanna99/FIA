### Finding #2 (chiuso) · Onboarding tour 10-step dismissibile via localStorage

**Risolto**. Chiave reale: `feapro-onboarding-seen-v4 = "skipped"` (vedi `OnboardingTour.tsx:27`). `addInitScript` in `beforeEach` setta la chiave prima del primo `page.goto`. Tour non appare nelle screenshot post-fix.

### Finding #3 (nuovo, scope v2.5.6 limitato) · TopBar Run button NON migrato a FeatureButton

**Pattern attuale** (`frontend/src/components/shell/TopBar.tsx:211-220`):

```tsx
{!isSolveOpen && (
  <Tooltip content={...}>
    <Button
      variant="run"
      disabled={!model || isRunning}
      onClick={() => model && run()}
    >
      Esegui
    </Button>
  </Tooltip>
)}
```

NON usa `FeatureButton` né `data-feature-id="run-static"`. Disabled solo su `!model || isRunning` — niente check precondizioni vincoli/carichi.

**Condizione di rendering**: `!isSolveOpen`. Quindi:
- **Default** (SolvePanel chiuso): topbar Run visibile, no `data-feature-id`
- **SolvePanel aperto**: topbar Run sparisce, SolvePanel interno Run con `data-feature-id="run-static"` appare

Il test E2E cerca `data-feature-id="run-static"` direttamente senza aprire SolvePanel → element not found.

**Decisione di scope v2.5.6 documentata** in `docs/v2.5.6-cluster-cf-state-preconditions_report.md` §"Decisioni di scope": "TopBar undo/redo skip migrazione, Button primitive non supporta size 28px custom". **MA**: la decisione era per undo/redo, NON per il Run button principale. Il Run button TopBar avrebbe potuto essere migrato (non è icon-only 28px), ed è il principale entry-point user-facing per "Esegui analisi".

**Implicazione UX Paolo**: l'utente che clicca "Esegui" in TopBar su modello vuoto vede ancora errore reattivo (backend "missing_constraints") invece di disabled + tooltip preventivo. Il fix BUG-027 v2.5.6 si applica solo dentro SolvePanel.

**Opzione fix (out of scope di questo brief)**: migrazione TopBar Run a `<FeatureButton featureId="run-${analysisType}">` o equivalente. ~15 righe. Vedi finding aperti.

### Finding #4 (nuovo) · `dashboard-action-template` click non apre TemplateGalleryDialog (visibile)

**Evidenza**: screenshot `02-template-dialog.png` scattata immediatamente post-click mostra Dashboard **virtualmente identica** a screenshot 01 (stessa hero, stesse quick action, stesso toast "Benvenuto"). Nessun overlay dialog visibile.

**Spiegazioni candidate** (da indagare):
1. **Race timing screenshot vs animation**: il dialog si apre con animazione (modal sliding/fade-in), screenshot scattata troppo presto. Improbabile: Playwright `await page.click()` di norma attende fino a quando il button è stable + propaga.
2. **`onClick={openTemplateGallery}` non dispatcha effettivamente l'event**: Dashboard.tsx:227 chiama `openTemplateGallery` (funzione locale), che probabilmente fa `window.dispatchEvent(new Event("feapro:open-template-gallery"))`. Se il listener in App.tsx è registrato dopo il click (timing), l'event va perso.
3. **Il dialog si apre E chiude immediatamente** per qualche logica downstream (es. template gallery vuota → auto-dismiss).

Il `waitForSelector` per "5 nodi" successivo ha comunque matchato qualcosa (l'app non blocca lo step), MA screenshot 03 mostra "Nessun modello" in topbar. Quindi il modello non è stato caricato — il match è stato un falso positivo (testo "5 nodi" preso da qualche tooltip o anteprima template, NON da un modello attivo).

**Out of scope test E2E**: fix nel sorgente Dashboard.tsx o test (es. usare `feapro:open-template-gallery` dispatch diretto invece di click).

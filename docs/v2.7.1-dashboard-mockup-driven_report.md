## 5. Lessons learned (per memoria futura)

### 5.1 PM (Claude Chat / brief)

- **Smettere di promettere percentuali match** nei rollup report. "~95% in
  attesa di smoke visivo" è disonesto se il gate non viene mai applicato.
- **Pre-brief**: per ogni nuovo brief Phase 4-6, eseguire `visual-audit.mjs`
  per identificare il gap reale tra mockup e live, NON stimare a occhio.
- **Smoke visivo manuale** non è più necessario nel formato 46/46 checkpoint
  da apporre a manina: il visual audit produce screenshot side-by-side
  oggettivi che Federico può navigare in <2 minuti.

### 5.2 Code (Claude Code / implementazione)

- **Pattern mockup-driven solid**: copia CSS verbatim → namespace sotto
  wrapper class (`.dash`, `.auth-shell`) → traduci HTML in JSX mantenendo
  class names → SVG verbatim camelCased per React.
- **Drop-in alias import** è il pattern più sicuro per sostituire
  componenti senza toccare i call site.
- **Architettura full-screen vs wrapped**: alcuni mockup sono pagine
  full-screen che sostituiscono il chrome (Auth, Dashboard); altri sono
  contenuto interno (workspaces dentro Shell custom). Sempre verificare
  con visual audit DOPO il primo deploy per identificare doppi chrome.

### 5.3 Workflow operativo Phase 4-6

Per ogni nuovo brief `v2.7.X` adesso:

1. **Pre-brief**: `node scripts/visual-audit.mjs` → identifica gap reale
2. **Brief Claude Chat** scope ristretto (1-2 mockup HTML per volta)
3. **Implementazione** Claude Code:
   - Leggi mockup HTML + CSS dedicato
   - namespace CSS sotto wrapper class
   - Componente React mockup-faithful
   - Drop-in integration (single-line se possibile)
4. **Quality gate intermedio**: tsc + vitest
5. **Visual audit re-run** PRIMA del rollup tag → screenshot probe
6. **Hotfix architecture** se chrome legacy/Shell custom interferisce
7. **Rollup tag + push + deploy**
8. **Visual audit re-run live** post-deploy per confermare

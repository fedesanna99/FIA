## Rischi identificati

| Rischio | Impatto | Mitigazione |
|---------|---------|-------------|
| `flyctl` auth potenzialmente scaduta (warning context canceled) | Deploy fallisce con errore auth | `fly auth whoami` o `fly auth login` prima del deploy |
| Secrets non documentati nel `[env]` (es. `JWT_SECRET`, OAuth keys) | Backend boot fail / login broken | Verificare con `fly secrets list -a fea-pro` prima del deploy. Se serve set: `fly secrets set KEY=value -a fea-pro`. Vedi `backend/auth/` per le chiavi richieste. |
| Volume `fea_data` non mappato correttamente | I modelli salvati spariscono dopo il deploy | `fly volumes list -a fea-pro` verificare `fea_data` esista in `fra`. Se assente: `fly volumes create fea_data --region fra --size 1` |
| Build OOM su Fly remote builder (numpy/scipy wheels pesanti) | Deploy fallisce metĂ  build | Fallback `fly deploy --local-only`. Workstation deve avere Docker installato. |
| WebSocket disconnect dopo deploy | Job progress streaming rotto | GiĂ  mitigato in fly.toml. Se reappare: `[http_service] idle_timeout = "300s"` |
| Cold-start lento (auto-stop attivo) | Prima request post-idle ~1-2s lenta | Self-ping giĂ  attivo (`FEAPRO_SELF_PING_*`). Per sessione Paolo, considerare `min_machines_running = 1` temporanea (`fly scale count 1 --max-per-region 1`) per evitare cold-start durante demo live. |
| ~80MB di chunk JS (three.js + index) | Prima load lenta su connessione mobile | Storico, giĂ  accettato. Service worker / preload non in scope. |

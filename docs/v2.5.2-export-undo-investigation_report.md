## Findings principali

### Bug #2 · Export PDF/XLSX

L'audit v2.3.2 disse "PDF/XLSX UI 3 path tutti morti — backend OK, dispatcher frontend rotto". La diagnosi tecnica è **diversa**:

1. **Backend non è coinvolto** nei 3 path UI. La generazione PDF è interamente client-side via jsPDF (`frontend/src/utils/reportPdf.ts:13` `generateReport(...)`). La generazione XLSX è client-side via SheetJS (`frontend/src/utils/exportXlsx.ts` `exportModelToXlsx`, audit-fix B6 v2.2.1). Gli endpoint backend (`backend/api/routes/io.py POST /api/io/export/*`) servono solo il pannello `ExportServerPanel` ("Tools · Export server"), non i 3 path mainstream.
2. **Path A** (button `ResultsOverviewCard.tsx:143` "Genera report PDF") e **Path B** (palette `wiz-report` `paletteItems.ts:277`) condividono lo stesso flow: dispatch `feapro:open-export-pdf` → listener `App.tsx:291` → `setReportExportOpen(true)` → `<ReportExportDialog>` modale → button "Scarica PDF" → `generateReport(...)`. Tutti i passaggi sono in place. **Probabilmente funzionanti** oggi, da verificare con repro live.
3. **Path C** (palette `exp-pdf-full` `paletteItems.ts:332` → `CommandPalette.tsx:255-272` case `quick-export.pdf`) chiama `generateReport(...)` diretto **senza toast di feedback**. L'utente vede "palette si chiude, nulla altro" → percezione "morto" anche se il file viene scaricato. **Fix: 1 riga di toast.**
4. **Sub-bug XLSX nascosto**: la voce palette `exp-xlsx` (`paletteItems.ts:333` → `CommandPalette.tsx:273-276`) chiama `exportDisplacementsCSV` + `exportModesCSV` (CSV piatti!) **invece** del corretto `exportModelToXlsx` (audit-fix B6). Il toast è esplicito ("scaricati CSV piatti") ma misleading. **Fix: 10 righe per replicare il pattern di `ExportView.doExcel:42-63`.**

### Bug #9 · Undo/Redo

L'audit v2.3.2 disse "v2.3.0 funziona nei test unitari ma non integrata con UI dialog". La diagnosi è:

1. **Il wiring dialog → store è corretto**. NodeDialog.tsx:100 chiama `addNode(n)` su `modelStore`, che esegue `applyModelAndPush` → `history.push(newModel)`. ✓
2. **Il problema è `qc.invalidateQueries(["model", model?.id])` in `NodeDialog.tsx:103` (e 6 altri dialog)**. Pipeline distruttiva:
   - mutation backend OK → `addNode(n)` → history.push ✓ history = `[baseline, +node]`
   - `invalidateQueries` → TanStack Query refetch via `useLoadModel.ts:14-29` → `setModel(refetched)` → `modelStore.ts:82` `history.clear()` + `history.push(refetched)` → history = `[refetched]` (1 solo elemento) ← **WIPE**
   - `canUndo: () => past.length > 1` → false → button disabled, Ctrl+Z no-op.
3. **7 dialog impattati**: NodeDialog, ElementDialog, ConstraintDialog, LoadDialog, ApplyClimateLoadsDialog, MeshWizardDialog, EditModelDialog. Più caller `applyMaterialToSelection` / `applySectionToSelection` da CommandPalette.
4. **Fix raccomandato (A)**: 15 righe in `modelStore.setModel` per preservare la history quando `m.id === current.id` (refetch dello stesso modello). Tutti i caller automaticamente fixati senza toccare i dialog.

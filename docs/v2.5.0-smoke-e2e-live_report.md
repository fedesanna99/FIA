### Finding #1 (BLOCKER per Paolo) · Il deploy live è v2.3.x, NON v2.5.0-paolo-ready

**Evidenza diretta** dalla screenshot `03-template-loaded.png`:

- **Topbar** mostra `F FEA Pro Free V2.3` (tag versione)
- **Footer** mostra `v2.3`
- Il bottone Run nel topbar è il vecchio `Esegui` (verde, design v2.3), senza alcun `data-feature-id` attribute
- I `data-feature-id="*"` (introdotti in v2.5.6 cluster F + v2.5.7 cluster A) **non esistono** nel DOM live

**Conclusione**: l'host `fea-pro.fly.dev` serve attualmente una **versione di build pre-pipeline UX 5-settimane**. Tutta la pipeline v2.5.1 → v2.5.9 (commit `282336e`) **non è stata deployata**. Il pre-deploy lookup (`docs/v2.5.0-pre-deploy-lookup.md`) era documentazione preparatoria, **non un `fly deploy` effettivo**.

**Implicazione critica per Paolo**: tutti i fix UX della pipeline (BUG-027 run senza precondizioni, BUG-014+015 toggle View, BUG-016+017 MakePanel italiano, BUG-039 ModelStatsBadge, ScaleIndicator dinamico, FeatureButton + EmptyState pattern, toastApiError handler globali, ecc.) **non sono visibili** all'utente che apre `https://fea-pro.fly.dev/` oggi. La sessione Paolo su questo deploy mostrerebbe la v2.3 con i bug pre-fix.

**Fix richiesto**: `fly deploy` da workstation di Federico prima della sessione. Vedi `docs/v2.5.0-pre-deploy-lookup.md` § "Raccomandazione brief deploy" per il pattern.

### Finding #2 (importante anche post-deploy) · Onboarding tour 10-step blocca smoke E2E

**Evidenza** dalla screenshot `03-template-loaded.png`: dopo il login + template loading, un **dialog modale "STEP 1/10 · Benvenuto in FEA Pro"** appare al centro dello schermo e blocca l'interazione con i pannelli sottostanti.

**Implicazione per il test E2E**: anche dopo il deploy della pipeline corretta, il test fallirebbe perché il `[data-feature-id="run-static"]` (visibile in v2.5.6+) è coperto dal modal onboarding. Il test deve **dismissare il tour** prima di procedere (button "Salta" visibile nell'angolo basso sinistra del modal).

**Fix per il test** (post-deploy v2.5.0):
- aggiungere dopo `await login(page)`: `await page.getByRole("button", { name: /^salta$/i }).click().catch(() => {});` (il `.catch` perché l'utente potrebbe aver già completato/salto il tour in sessione precedente)
- O più robusto: localStorage seed che setta il flag onboarding completato prima del primo `page.goto`

**Out of scope di questo brief** (T3 STOP rule #4: documento, NO fix). Lo aggiungeremo nello sprint smoke E2E successivo POST-deploy.

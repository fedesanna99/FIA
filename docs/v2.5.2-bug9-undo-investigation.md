# Bug #9 · Undo/Redo · Investigation report

**Data**: 2026-05-26
**Scope**: mappatura tecnica del wiring incompleto fra dialog di mutation e `useModelHistory`, come dichiarato nell'audit v2.3.2 ("Undo/Redo broken in produzione · dopo `addNode`, sia Ctrl+Z che button 'Annulla' non rimuovono il nodo. La feature v2.3.0 funziona nei test unitari ma non integrata con UI dialog").
**Esecutore**: Claude Code

## TL;DR

**Root cause individuata**: la history NON è broken nei dialog. La history è **wipe-ata dal refetch TanStack Query** dopo ogni mutation.

Pipeline completa:

1. NodeDialog · mutation `useMutation` chiama `modelsApi.addNode(model.id, payload)` (POST backend)
2. `onSuccess(n)` chiama `useModelStore.addNode(n)` (modelStore.ts:109) → `applyModelAndPush` → `history.push(new_model)` ✓ history popolata
3. `onSuccess` chiama anche `qc.invalidateQueries({ queryKey: ["model", model?.id] })` (NodeDialog.tsx:103)
4. TanStack Query invalida → triggera refetch su `useLoadModel` (hooks/useModel.ts:14-29)
5. `useEffect` su `query.data` → chiama `setModel(query.data)` (useModel.ts:24)
6. **`setModel` in `modelStore.ts:80-83`** esegue `history.clear()` **e** `history.push(refetched_model)` — la history viene **azzerata** a 1 solo elemento (il refetched)
7. `canUndo: () => past.length > 1` (historyStore.ts:44) → **false**. Button undo disabled. Ctrl+Z anche se invocato fallisce: `undo()` ritorna null (`past.slice(0,-1) = []`).

Il problema è **architetturale** in `setModel`, NON nei dialog. I dialog sono cablati correttamente alla store mutation. Il fix è 1 riga in `modelStore.ts` per preservare la history quando il refetch è dello stesso model.id.

## 1. Store e history (asset esistente, NON da toccare)

`frontend/src/store/historyStore.ts` (96 righe, factory `createHistoryStore<T>(limit = 50)`):

- Espone: `past[]`, `future[]`, `limit`, `canUndo()`, `canRedo()`, `push(snapshot)`, `undo()`, `redo()`, `clear()`, `setLimit(n)`.
- `canUndo: () => get().past.length > 1` (riga 44) — richiede ≥ 2 snapshot, baseline + ≥ 1 mutation.
- `push(snapshot)`: trim a `limit` snapshot, svuota `future`.
- `undo()`: sposta `past.last` in `future`, ritorna l'ultimo rimasto in `past` (o null se vuoto).
- `redo()`: sposta `future.first` in `past`, lo ritorna.
- Istanza pre-creata: `useModelHistory = createHistoryStore<unknown>(50)` (riga 95).

`frontend/src/store/modelStore.ts` (179 righe) integra:

- Helper privato `applyModelAndPush(set, newModel)` (riga 62-68): `set({model, lastSavedAt})` + `useModelHistory.getState().push(newModel)`.
- Tutte le 12 mutations chiamano `applyModelAndPush`: `addNode`, `updateNode`, `removeNode`, `addElement`, `updateElement`, `removeElement`, `addLoad`, `updateLoad`, `removeLoad`, `addConstraint`, `updateConstraint`, `removeConstraint` (righe 109-164).
- `setModel(m)` (riga 72-84) inizializza una **nuova** history: `history.clear()` + `history.push(m)` ← **il punto critico del bug**.
- `undo()` (riga 166-171): legge `useModelHistory.undo()`, applica `set({model: prev, lastSavedAt: new Date()})`.
- `redo()` (riga 172-177): simmetrico.

L'API store è **corretta e ben disegnata**. Test unitari `modelStore.test.ts` e `historyStore.test.ts` passano (584 + 13 = 597/597 in vitest dopo S0).

## 3. Caller history-aware funzionanti

I caller che **funzionano** con la history (= Ctrl+Z fa undo reale) sono quelli che:

- Modificano il modello tramite le mutation di `modelStore` (`addNode`, `addElement`, ecc.)
- **NON** triggerano un refetch via TanStack Query subito dopo.

Esempi:

- `ClickPlane.tsx:31` (`addNode(n)` dopo creazione viewport-click): chiama `modelsApi.addNode` + `addNode(n)` ma **dopo** un `useMutation` che non invalida `["model", id]` (verificare). Se il pattern è "POST backend → store.addNode → niente invalidate", la history sopravvive.
- `useModelStore.getState().undo()` / `.redo()` stessi: agiscono direttamente sullo store, niente refetch in mezzo.

Il pattern funzionante è "store-only mutation, no refetch". Ma è incompatibile con le esigenze TanStack Query (cache consistency).

## 4. Discrepanza L2 falso positivo

Riferimento: `docs/v2_3_4_quality_checkpoint/L2_functional_flows_report.md:51` segna ✅ il test `happy: Ctrl+Z dopo modifica → state revertito`. Lookup v2.4.9 aveva già notato che è falso positivo (probabilmente Ctrl+Z senza una mutation reale prima, quindi "niente da revertire" = "nessun cambiamento" = test PASS senza coprire il bug vero).

Conferma alla luce di T2: l'unico modo per il test L2 di registrare PASS è che la mutation di test non passi attraverso `useMutation` con `invalidateQueries`. Esempio plausibile: il test usa `useModelStore.getState().addNode(...)` direttamente (store-only, no refetch) per simulare la modifica, poi preme Ctrl+Z e verifica che `model.nodes` torni allo stato precedente. Ma il flusso "user-facing" passa SEMPRE attraverso il dialog (con refetch), quindi il test L2 non lo esercita.

Va riscritto nel brief quality automation finale (`v2.5.10-paolo-readiness-gate`) con il pattern: `open NodeDialog → submit form → wait mutation completion → wait refetch → Ctrl+Z → assert node removed`. Solo così il test fallirebbe sul bug vero.

## 5. Ipotesi di fix (NO implementazione in questo brief)

Tre opzioni, in ordine di chirurgicità crescente:

### Fix A — Preserva history nel `setModel` se è lo stesso model.id (raccomandato)

`frontend/src/store/modelStore.ts:72-84`, modificare `setModel`:

```ts
setModel: (m) => {
  const current = get().model;
  const isSameModel = m && current && m.id === current.id;
  set({
    model: m,
    selectedNodeIds: new Set(),
    selectedElementIds: new Set(),
    lastSavedAt: m ? new Date() : null,
  });
  const history = useModelHistory.getState();
  if (!isSameModel) {
    // Nuovo modello (o reset): inizializza la history.
    history.clear();
    if (m) history.push(m);
  } else if (m && history.past.length > 0) {
    // Stesso model.id: refetch post-mutation. La history è già popolata
    // dalla mutation che ha preceduto. Pusha solo se il refetched diverge
    // dall'ultimo snapshot (defensive: backend potrebbe aver normalizzato).
    const last = history.past[history.past.length - 1];
    if (JSON.stringify(last) !== JSON.stringify(m)) {
      history.push(m);
    }
  }
},
```

**Effetto**: per il flow `addNode via NodeDialog → invalidateQueries → refetch → setModel(refetched_same_id)` la history NON viene wipata. Ctrl+Z funziona.

**Scope**: 1 file (`modelStore.ts`), ~15 righe di modifica. Tutti i 7 dialog + i 2 caller palette `apply-material/section` automaticamente fixati.

**Rischi**: il `JSON.stringify` comparison può essere costoso su modelli grossi (>10k entities). Si può usare un hash più rapido (es. solo length-based: `last.nodes.length !== m.nodes.length || ...`).

**Test da aggiungere**: 
- `modelStore.test.ts`: nuovo test "setModel con stesso id preserva history"
- `modelStore.test.ts`: nuovo test "setModel con id diverso wipe history"

### Fix B — Skippare `invalidateQueries` post-mutation se lo store è già aggiornato

Rimuovere `qc.invalidateQueries({ queryKey: ["model", model?.id] })` da tutti i 7 dialog: lo store è già la single source of truth post-mutation (modelStore è aggiornato in `onSuccess`).

**Scope**: 7 file, ~7 righe (1 per dialog).

**Rischi**: la cache TanStack Query del modello diverge dal server. Se in qualche altro punto del codice viene letto `query.data` (es. un secondo `useLoadModel` montato in parallelo), riceverà il vecchio modello. Possibile data corruption silente. **Sconsigliato.**

### Fix C — Refactor history per dedup snapshot identici a livello `push`

Modificare `historyStore.ts.push(snapshot)` per skipparlo se il nuovo snapshot è identico all'ultimo in `past`. Più clean architetturalmente ma richiede una `equals` function generica.

**Scope**: 1 file, ma cambia il contratto di `push` (potenziali effetti su altri caller). Più rischioso, da fare a parte.

**Raccomandazione**: **Fix A** per v2.5.4. È il fix più chirurgico (1 file, 15 righe) e tutti i 7 dialog + i palette caller automaticamente funzionano. Scope (a) del brief: **0.5 giorni**.

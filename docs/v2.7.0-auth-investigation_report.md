## 2. Mapping routes attuali → target brief

| Target brief | Attuale | Gap |
|---|---|---|
| `/login` route + LoginPage | Mode `"login"` in AuthScreen tab | No URL, mode interno |
| `/signup` route + SignupPage | Mode `"register"` in AuthScreen tab | No URL, mode interno, 2 campi vs 5 |
| `/forgot-password` route + ForgotPasswordPage | **NON ESISTE** | Page nuova, no entrypoint |
| `/verify-email` route + EmailVerifyPage | **NON ESISTE** | Page nuova, no entrypoint, no backend |

**Conseguenza**: introdurre React Router significherebbe:
- Aggiungere `react-router-dom` dipendenza
- Wrappare `<App>` o sotto-tree in `<BrowserRouter>` (modifica strutturale)
- Convertire AuthGate per coabitare con router (es. `<ProtectedRoute>` o `<Outlet>` + redirect logic)
- Aggiornare `main.tsx` (entry) o `App.tsx` per montare router
- Migrare ogni navigation pattern futuro (Phase 4.2 Dashboard, 4.3 Templates, 5.x Studio panels) verso route URL-based

**Impact stimato modifica strutturale**: NON solo lo scope auth v2.7.0 — è una decisione architetturale che influenza tutta Phase 4-6.

## 3. Shape API endpoint vs form mockup

### LOGIN (state 1)

Mockup campi: `email`, `password`, `stayLogged` (checkbox 30 giorni).

| Campo | Backend supporto |
|---|---|
| `email` | ✅ `/api/auth/login` |
| `password` | ✅ `/api/auth/login` |
| `stayLogged` (30 giorni) | ❌ Backend non gestisce remember-me; token JWT ha TTL fisso |

**Gap minore**: `stayLogged` checkbox è cosmetico. Implementarlo richiede backend remember-me (modifica auth_settings.JWT_TTL). Decisione possibile: **mantenere checkbox UI ma ignorato server-side per v2.7.0** (toast info "remember-me arriva in v2.8") oppure rimuovere checkbox dal form (diverge dal mockup ma evita misleading UX).

### SIGNUP (state 2)

Mockup campi: `nome`, `cognome`, `email`, `password`, `ruolo` (select 5 options), `acceptedTerms` (checkbox required).

| Campo | Backend supporto |
|---|---|
| `email` | ✅ `/api/auth/register` |
| `password` | ✅ `/api/auth/register` |
| `nome`, `cognome` | ❌ Non in payload, non in user model |
| `ruolo` (ingegnere/architetto/docente/studente/altro) | ❌ Non in payload, non in user model |
| `acceptedTerms` | ❌ Backend non traccia consenso TOS |

**Gap critico**: il mockup signup raccoglie 5 campi business-relevant (profilo + ruolo + consenso) che backend ignora. Opzioni:

| Opzione | Pro | Contro |
|---|---|---|
| **A · Solo UI** (collect ma drop, log warning) | Zero scope backend, mockup-fidele visuale | UX ingannevole, dati persi |
| **B · Backend extension** (`POST /api/auth/register` esteso) | UX onesta, dati salvati | Out-of-scope brief, ~2-3h aggiuntive |
| **C · Form ridotto** (rimuovi campi non-supportati) | UX onesta, no scope creep | Diverge dal mockup (regola d'oro violata) |

### FORGOT (state 3)

Mockup: `email` → invio link reset. Backend **non ha endpoint**.

| Opzione | Note |
|---|---|
| **A · Mock UI** (form + submit → toast "Email inviata" senza backend call) | Mockup-fidele ma no funzionalità |
| **B · Backend extension** (`POST /api/auth/forgot-password` + email service) | ~3-4h, richiede SMTP setup |
| **C · CTA "in arrivo v2.8"** | Form readonly o disabled |

### VERIFY (state 4)

Mockup: OTP 6 cifre + resend countdown 60s. Backend **non ha endpoint verify-email né flow di verifica**.

Inoltre AuthUser shape NON ha `email_verified: boolean` — l'utente è considerato autenticato immediatamente dopo `register`, no email confirmation step.

| Opzione | Note |
|---|---|
| **A · Mock UI** (UI OTP completa, submit OTP → toast "Verificato" senza backend) | Mockup-fidele ma totalmente mock |
| **B · Backend extension** (email verify flow + `email_verified` campo + redirect post-register) | ~4-5h, modifica register flow, modifica AuthGate (block !email_verified?) |
| **C · Rimuovi state verify** | Drop scope mockup per allineamento backend |

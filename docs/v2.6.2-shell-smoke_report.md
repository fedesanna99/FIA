### Ώθγρ Finding (alto interesse Federico): coesistenza Shell nuova + ViewportHud legacy

**Problema**: anche nella modalit├ι nuova Shell (post-template, desktop,
!focus), **il vecchio `ViewportHud.tsx` continua a renderizzarsi sopra il
viewport** insieme alla `ViewportCanvasTabs`.

Causa architetturale:
- In `App.tsx` v2.6.2, quando `useNewShell` ├ρ true, viene reso:
  ```tsx
  <Shell>
    <div className="absolute top-0 left-0 right-0 z-10">
      <ViewportCanvasTabs ... />   // LEGACY tabs
    </div>
    <div className="absolute inset-0 pt-[36px]">
      <Viewport3D />                // Wrapper R3F che internamente monta ViewportHud
    </div>
  </Shell>
  ```
- `Viewport3D.tsx` (non toccato in Fase 2 per design έΑΦ out of scope) include
  `<ViewportHud />` legacy come componente figlio.
- Risultato: i 3 chip legacy (Telaio portale, N nodi, Tecnica) appaiono
  ancora nell'area top-left, accanto/sotto ai nuovi HUD shell.

**Conseguenza visiva**:
- Dichiarazione "card overlap top-left RISOLTO" ├ρ **prematura**. Il chip
  vecchio ├ρ ancora l├υ, anche se ora il bug overlap originale (chip che
  coprivano un badge blu sotto) potrebbe essere ridotto perch├σ la nuova
  Shell aggiunge nuovi HUD floating senza confliggere con i chip in
  posizione.
- Federico aprendo lo screenshot `04-viewport-top-left.png` vedr├ι sia i
  3 chip legacy che eventuali nuovi HUD floating del nuovo design system.

**Fix necessario in Fase 3**:
1. Rimuovere `<ViewportHud />` dal rendering di `Viewport3D.tsx` quando la
   nuova shell ├ρ attiva, oppure
2. Rifattorizzare `Viewport3D.tsx` per non includere HUD interni (i HUD
   diventano scope della Shell, separati dal Canvas R3F)
3. Rimuovere `<ViewportCanvasTabs />` dal rendering App.tsx in modalit├ι
   nuova Shell (i 5 tab sono gi├ι nella ShellPanel come Radix Tabs)

**Severit├ι**: P2 (cosmetic/incomplete UX). Non bloccante per Fase 3 perch├σ
i due sistemi visivi coesistono senza errori funzionali έΑΦ i bottoni
funzionano, i dati sono corretti. ├Ι un "doppione visivo" da pulire.

## Bug emersi / Finding

### F1 (P2) ┬╖ Coesistenza Shell nuova + ViewportHud legacy

Vedi sezione "Overlap audit" sopra. Da fixare in Fase 3 togliendo ViewportHud
e ViewportCanvasTabs dal codepath `useNewShell=true` di App.tsx.

### F2 (P2) ┬╖ Path "Export PDF" nella nuova palette non matcha registry legacy

`ShellCommandPalette.tsx` (Fase 2) ha 4 voci hardcoded (AI ask, Esporta
report PDF, Vai a Modello/Analisi/Verifiche, Esegui statica, Annulla).
Il registry esistente `paletteItems.ts` (~140 voci) NON ├ρ connesso alla
nuova palette.

In Fase 5 (migration unificata) il registry esistente deve fluire nella
nuova UI cmdk. Per ora nella shell nuova l'utente vede solo 7 voci, non
le 140 originali.

Impatto: discoverability ridotta in modalit├ι nuova Shell. Pipeline UX
funziona (i registry items sono ancora invocabili tramite handler diretti,
shortcut, dispatch event), ma la palette nuova ├ρ un placeholder visivo.

### F3 (P3 nice-to-have) ┬╖ `tb-model-id` mostra "EX_POR" invece di "TPF" o simile

Il code in `ShellTopBar.tsx`:
```tsx
{model?.id && <span className="tb-model-id">{model.id.slice(0, 6).toUpperCase()}</span>}
```

Mostra i primi 6 caratteri dell'UUID modello in maiuscolo (`EX_PORTAL` έΗΤ
`EX_POR`). Il mockup mostrava un ID semantico tipo "UC1". Per Fase 3 si
pu├▓ aggiungere una mappa template-id έΗΤ short-id, o nascondere il
`.tb-model-id` se non c'├ρ metadata semantica.

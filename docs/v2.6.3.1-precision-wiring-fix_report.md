## T2 · Fix ChecksDetailTable wiring (BUG-AUDIT-#1)

### Root cause scoperto

Inizialmente sospettato: VerifyPanel dentro ShellPanel 380px hardcoded.
**Più profondo**: anche se Shell renderizzasse VerifyPanel in area piena,
il wrapper interno `PanelChrome` ha **width fisse hardcoded**:

```tsx
"w-full md:w-[300px] lg:w-[340px] xl:w-[380px]"
```

VerifyPanel passa attraverso PanelChrome che lo comprime a 380px max.

### Fix

3 modifiche puntuali (override autorizzazioni esplicite brief +
PanelChrome aggiunto al perimetro):

| File | Diff | Cosa |

| `frontend/src/shell/panels/VerifyPanel.tsx` | +14 / -2 | Prop `fullArea?: boolean` forwarded a `PanelChrome.fullWidth` |

### Trade-off documentato

Canvas R3F è **unmount-remount** sul cambio workspace verso/da `verifiche`.
Accettabile perché:
- In Verifiche l'utente non vede il viewport 3D (è una tabella)
- Cambio workspace è raro (una volta per minuti)
- Stato modello/results vive in Zustand (no perdita reale di state)

## Note di metodo (lesson learned)

Il brief v2.6.3.1 si è dimostrato chirurgico e diagnosticamente corretto:

1. **Identificazione precisa del root cause** (BUG-#1 ShellPanel 380px + PanelChrome width fisse): un solo file Shell.tsx non bastava, doveva propagarsi a PanelChrome → VerifyPanel
2. **Strategia ibrida T3** (PercorsoFullScreenDemo come modello) > Opzione A workspace ID dedicato: stesso risultato visivo, no refactor di workspaceStore
3. **Trade-off Canvas R3F unmount** accettabile: documentato esplicitamente, no sorprese
4. **0 backend changes**: tutti i fix sono UI/orchestration

Il fix architettonico vero (PanelChrome `fullWidth` prop) è generalizzabile: in futuro, ogni nuovo workspace che necessita area piena può usare `VIEWPORT_TAKEOVER_WORKSPACES` + `fullArea` pattern.
